# 01 협력
- 객체지향의 세계에서 기능을 구현할수 있는 유일한 방법이다.
- 두 객체 사이의 협력은 하나의 객체가 다른 객체에게 도움을 요청할 때 시작된다.
- 메시지 전송은 객체 사이의 협력을 위해 사용할 수 있는 유일한 커뮤니케이션 수단이다.
- 협력은 객체를 설계하는 데 필용한 일종의 문맥을 제공한다.

# 02 책임
- 객체에게 얼마나 적절한 책임을 할당하느냐가 설계의 전체적인 품질을 결정한다.
## CRC 카드
- 후보(Candidate), 책임(Responsibility), 협력자(Collaraborator)
- 후보는 클래스, 객체, 컴포넌트, 역할 어떤 것도 될 수 있다.
- 카드의 선이 없는 면에는 후보의 목적을 기술한다.
- 다른 면 좌측 하단에는 목적을 좀 더 세분화해서 무엇을 알고 무엇을 해야하는지에 대한 책임을 차례대로 적는다.
- 카드 우측에는 책임을 수행하면서 함께 협력할 협력자들을 나열한다.

## 책임 주도 설계
- 책임을 찾고 수행할 적절한 객체를 찾아 책임을 할당하는 방시으로 협력을 설계하는 방법

## 메시지가 객체를 선택하게 해야 하는 이유
- 객체가 최소한의 인터페이스를 가질 수 있게 된다.
- 객체는 추성적인 인터페이스를 가질 수 있게 된다.

# 역할
- 객체가 어떤 특정한 협력 안에서 수행하는 책임의 집합

- 책임의 관점에서 두 협력을 바라보면 AmountDiscountPolicy와 PercentDiscountPolicy 모두 할인 요금 계산이라는 동일한 책임을 수행한다.
- 역할은 두 종류의 구체적인 객체를 포괄하는 추상화이다.
- 역할을 이용하면 불필요한 중복 코드를 제거할 수 있다.

## 추상 클래스와 인터페이스
- 역할을 정의할 수 있는 구현 방법
- 추상 클래스 : 추상 클래스는 책임의 일부를 구현해 놓은 것
- 인터페이스 : 일체의 구현 없이 책임의 집합을 나열해 놓은 것

## 객체 대 역할
- 동일한 종류의 객체가 하나의 역할을 항상 수행한다면 둘은 동일한 것
- 어떤 협력에서 하나 이상의 객체가 동일한 책임을 수행할 수 있다면 역할은 서로 다른 방법으로 실행할 수 있는 책임의 집합이 된다.
- 역할이란 프로그램이 실행될 때 소프트웨어 기계 장치에서 적절한 객체로 메워 넣을 수 있는 하나의 슬롯
