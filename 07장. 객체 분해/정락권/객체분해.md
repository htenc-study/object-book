# 오브젝트 7장. 객체 분해

## 목표
- 객체 분해가 왜 필요한지 설명할 수 있다.
- 기능 분해, 모듈, 추상 데이터 타입, 클래스 기반 객체지향의 차이를 비교할 수 있다.
- "변경의 축"에 따라 어떤 분해 방식을 선택해야 하는지 설명할 수 있다.

## 한 줄 요약
7장은 "클래스를 쓰면 곧 객체지향"이 아니라는 점을 보여준다. 핵심은 코드를 어떤 기준으로 분해하느냐이며, 기능 중심 분해에서 시작해 정보 은닉, 데이터 추상화, 다형성 기반 객체지향으로 발전하는 이유를 단계적으로 설명한다.

---

## 1. 추상화와 분해

### 왜 분해가 필요한가
- 현실 문제는 한 번에 다루기엔 너무 크고 복잡하다.
- 인간의 단기 기억은 한계가 있기 때문에 복잡한 문제를 그대로 처리하면 인지 과부하가 발생한다.
- 그래서 사람은 `추상화`와 `분해`를 사용한다.

### 핵심 개념
- `추상화`: 불필요한 세부를 제거하고 핵심만 남기는 작업
- `분해`: 큰 문제를 다룰 수 있는 작은 단위로 나누는 작업
- 분해의 단위는 단순한 조각이 아니라 의미 있는 논리적 덩어리다.

### 이 장의 두 가지 분해 방식

| 분해 방식 | 기준 | 대표 방식 |
| --- | --- | --- |
| 프로시저 추상화 | 무엇을 해야 하는가 | 기능 분해, 절차형 분해 |
| 데이터 추상화 | 무엇을 알고 있어야 하는가 | 추상 데이터 타입, 클래스 기반 객체지향 |

객체지향은 기능을 없애는 방법이 아니라, 변경에 덜 흔들리도록 분해 기준을 바꾸는 방법이다.

---

## 2. 프로시저 추상화와 기능 분해

### 프로시저란
- 반복되거나 유사한 작업을 하나의 이름 아래 모아 둔 추상화 단위다.
- 내부 구현을 몰라도 인터페이스만 알면 사용할 수 있다.
- 로직 재사용과 중복 제거에 효과적이다.

### 기능 분해의 전형적인 흐름
급여 관리 예제는 다음 흐름으로 시작한다.

1. 사용자에게 세율을 입력받는다.
2. 직원 급여를 계산한다.
3. 결과를 출력한다.

즉, 시스템을 "무엇이 존재하는가"보다 "어떤 순서로 실행되는가" 중심으로 본다.

### 하향식 접근법
- 가장 상위 기능을 정의한다.
- 그 기능을 하위 기능으로 계속 분해한다.
- 구현 가능한 수준까지 세분화한다.

예:
- `급여를 계산한다`
- `세율을 입력받는다`
- `직원 급여를 계산한다`
- `결과를 출력한다`

### 예제 A. 가장 단순한 기능 분해

`code/a_functional_decomposition/employees.rb`

```ruby
$employees = ["직원A", "직원B", "직원C"]
$basePays = [400, 300, 250]

def main(name)
  taxRate = getTaxRate()
  pay = calculatePayFor(name, taxRate)
  puts(describeResult(name, pay))
end
```

- `main`이 전체 흐름을 통제한다.
- 데이터는 전역 배열에 있다.
- 프로그램의 중심은 직원 객체가 아니라 급여 계산 절차다.

### 기능 분해의 장점
- 단순한 문제를 빠르게 구조화할 수 있다.
- 계산 절차가 명확한 프로그램에 적합하다.
- 알고리즘 설명이나 문서화에 유리하다.

### 기능 분해의 한계

#### 1. 메인 함수가 비대해진다
- 현실 시스템은 하나의 시나리오만 수행하지 않는다.
- 기능이 늘수록 메인 함수가 분기와 제어의 중심이 된다.

#### 2. UI와 비즈니스 로직이 결합된다
- 입력, 계산, 출력이 한 흐름에 섞인다.
- 계산 로직만 따로 재사용하거나 테스트하기 어렵다.

#### 3. 실행 순서를 너무 빨리 고정한다
- "무엇을 할 것인가"보다 "어떻게 실행할 것인가"에 먼저 묶인다.

#### 4. 데이터 변경의 파급효과가 크다
- 데이터 구조가 바뀌면 여러 함수가 함께 영향을 받는다.

#### 5. 새로운 개념이 추가될수록 조건문이 늘어난다
- 직원 유형이 늘어나면 분기 로직이 곳곳에 퍼진다.

### 예제 B. 기능이 추가되면 메인 함수가 분기점이 된다

`code/b_add_function/employees.rb`

```ruby
def main(operation, args={})
  case(operation)
  when :pay then calculatePay(args[:name])
  when :basePays then sumOfBasePays()
  end
end
```

- 급여 계산 외에 `총 기본급 계산` 기능이 추가됐다.
- 기능이 늘수록 `main`이 시스템의 진입점이자 분기점이 된다.

### 예제 C. 데이터 변경이 절차 전체로 퍼진다

`code/c_data_change/employees.rb`

```ruby
$hourlys = [false, false, false, true, true, true]
$timeCards = [0, 0, 0, 120, 120, 120]

def calculatePay(name)
  taxRate = getTaxRate()
  if (hourly?(name)) then
    pay = calculateHourlyPayFor(name, taxRate)
  else
    pay = calculatePayFor(name, taxRate)
  end
end
```

- 직원 타입 차이가 생기자 전역 데이터가 늘어난다.
- 동시에 여러 함수가 함께 수정된다.
- 즉, 변경의 축은 직원 타입인데 코드는 절차 중심으로 묶여 있다.

### 정리
기능 분해는 출발점으로는 좋지만, 시스템이 커지고 데이터와 타입 변화가 생기면 빠르게 경직된다.

## 3. 정보 은닉과 모듈

### 왜 모듈이 필요한가
기능 분해의 가장 큰 문제 중 하나는 데이터가 여기저기 노출된다는 점이다. 이를 줄이기 위한 핵심 원리가 `정보 은닉`이다.

### 정보 은닉의 핵심
- 외부에 꼭 필요한 인터페이스만 공개한다.
- 내부 데이터 구조와 계산 규칙은 감춘다.
- 시스템을 기능이 아니라 "감춰야 할 비밀"을 기준으로 나눈다.

### 모듈이 감추는 것
- 자주 바뀌는 데이터 구조
- 복잡한 계산 로직
- 외부가 몰라도 되는 내부 상태

### 예제 D. 비밀을 감추는 첫 단계

`code/d_module/employees.rb`

```ruby
module Employees
  def Employees.calculatePay(name, taxRate)
    if (Employees.hourly?(name)) then
      pay = Employees.calculateHourlyPayFor(name, taxRate)
    else
      pay = Employees.calculatePayFor(name, taxRate)
    end
  end
end
```

- 외부는 `Employees.calculatePay(...)`만 호출한다.
- 직원 데이터 구조와 계산 방식은 모듈 안으로 숨는다.
- 전역 데이터 노출은 줄었지만, 타입 분기 자체는 아직 남아 있다.

### 모듈의 장점
- 데이터 변경의 영향 범위를 좁힌다.
- 비즈니스 로직과 UI를 분리하기 쉬워진다.
- 이름 충돌과 전역 오염을 줄인다.

### 모듈의 한계
- 개별 직원 인스턴스를 자연스럽게 표현하지 못한다.
- 여전히 연산 중심 분해다.
- 새로운 타입이 추가되면 모듈 내부 여러 함수가 수정된다.

### 정리
모듈은 기능 분해보다 진전된 방식이지만, 타입별 행위 차이를 다루기에는 아직 부족하다.

## 4. 데이터 추상화와 추상 데이터 타입

### 데이터 추상화의 관점
프로시저 추상화가 "해야 할 일"을 중심으로 시스템을 나눈다면, 데이터 추상화는 "다뤄야 할 개념"을 중심으로 시스템을 나눈다.

### 타입이란
- 어떤 종류의 데이터를 저장할 수 있는가
- 그 데이터에 어떤 연산을 적용할 수 있는가

즉, 타입은 값의 집합과 오퍼레이션의 집합을 함께 의미한다.

### 추상 데이터 타입(ADT)의 조건
- 새로운 타입을 정의할 수 있어야 한다.
- 타입에 적용할 오퍼레이션을 정의할 수 있어야 한다.
- 외부는 공개된 오퍼레이션을 통해서만 접근해야 한다.
- 같은 타입의 인스턴스를 여러 개 만들 수 있어야 한다.

### 예제 E. 인스턴스는 생겼지만 타입 코드는 남아 있다

`code/e_abstract_data+type/employees.rb`

```ruby
Employee = Struct.new(:name, :basePay, :hourly, :timeCard) do
  def calculatePay(taxRate)
    if (hourly) then
      return calculateHourlyPay(taxRate)
    end
    return calculateSalariedPay(taxRate)
  end
end
```

- 이제 `Employee` 인스턴스를 여러 개 만들 수 있다.
- 데이터와 연산도 한 타입 안에 모였다.
- 하지만 `hourly` 플래그로 타입을 구분하므로 내부 조건문이 남아 있다.

### ADT의 장점
- 데이터와 관련 연산이 함께 묶인다.
- 내부 표현을 숨길 수 있다.
- 인스턴스 개념을 도입할 수 있다.

### ADT의 한계

#### 1. 타입 코드를 내부에 두기 쉽다
- `hourly`, `kind`, `type` 같은 필드가 생긴다.

#### 2. 오퍼레이션 중심으로 묶인다
- `calculatePay` 하나가 모든 직원 유형을 처리한다.
- 즉, 타입별 행위 차이를 한 메서드 안에서 통합한다.

#### 3. 새로운 타입 추가에 약하다
- 프리랜서 직원이 추가되면 기존 오퍼레이션들을 모두 수정해야 한다.

### 핵심 이해
- ADT는 데이터를 숨긴다.
- 하지만 타입별 행위 차이를 다형성으로 분산하진 못한다.
- 그래서 객체지향으로 가는 중간 단계로 볼 수 있다.

---

## 5. 클래스와 객체지향

### 클래스는 왜 ADT와 다른가
많은 경우 `클래스 = 추상 데이터 타입`처럼 생각하지만, 책은 둘을 구분한다. 차이는 `상속과 다형성`이다.

### 비교

| 구분 | 추상 데이터 타입 | 클래스 기반 객체지향 |
| --- | --- | --- |
| 타입 차이 처리 | 내부 조건문 | 다형성 |
| 중심 | 타입 추상화 | 절차 추상화 |
| 상속/다형성 | 없음 | 있음 |

### 예제 F. 타입 분기를 객체에게 위임

`code/f_class/employees.rb`

```ruby
class SalariedEmployee < Employee
  def calculatePay(taxRate)
    return basePay - (basePay * taxRate)
  end
end

class HourlyEmployee < Employee
  def calculatePay(taxRate)
    return (basePay * timeCard) - (basePay * timeCard) * taxRate
  end
end
```

- `if hourly` 같은 타입 판별 조건문이 사라진다.
- 외부는 `employee.calculatePay(taxRate)`만 호출하면 된다.
- 어떤 계산이 실행될지는 객체가 스스로 결정한다.

### 객체지향에서 달라지는 점

#### 1. 타입 검사 대신 메시지를 보낸다
- "너는 어떤 타입이니?"가 아니라 "이 메시지를 처리해줘"가 된다.

#### 2. 변경이 각 타입으로 분산된다
- 시급 직원 규칙은 `HourlyEmployee`
- 월급 직원 규칙은 `SalariedEmployee`

#### 3. 새로운 타입 추가가 쉬워진다
- `FreelancerEmployee`를 추가해도 기존 타입 코드를 크게 건드리지 않을 수 있다.
- 이것이 OCP에 가까운 구조다.

### 정리
객체지향의 핵심은 클래스를 쓰는 것이 아니라, 다형성으로 타입 차이를 분산시키는 데 있다.

## 6. 추상 데이터 타입 vs 클래스 기반 객체지향

| 구분 | 추상 데이터 타입 | 클래스 기반 객체지향 |
| --- | --- | --- |
| 묶는 기준 | 오퍼레이션 중심 | 타입 중심 |
| 타입 차이 처리 | 내부 조건문 | 다형성 |
| 새로운 타입 추가 | 어렵다 | 쉽다 |
| 새로운 오퍼레이션 추가 | 비교적 쉽다 | 상대적으로 어렵다 |
| 대표 성격 | 타입 추상화 | 절차 추상화 |

### 왜 이런 차이가 생기는가

#### ADT
- `calculatePay` 하나가 모든 직원 유형을 처리한다.
- 연산 추가는 상대적으로 쉽다.
- 대신 새로운 직원 유형이 생기면 기존 연산 전체를 수정해야 한다.

#### 객체지향
- 직원 유형별로 자기 행위를 가진다.
- 새로운 타입 추가는 쉽다.
- 대신 새로운 공통 오퍼레이션이 생기면 관련 타입을 함께 수정해야 할 수 있다.

무엇을 묶느냐가 설계의 본질이다. ADT는 같은 연산을 묶고, 객체지향은 같은 타입의 책임을 묶는다.

## 7. 변경을 기준으로 선택하라

### 타입 추가가 자주 일어나면
- 객체지향이 유리하다.
- 예: 직원 유형, 결제 수단, 할인 정책, 배송 정책

### 오퍼레이션 추가가 자주 일어나면
- ADT가 더 단순할 수 있다.
- 예: 조회, 통계, 출력, 변환 작업이 계속 늘어나는 경우

### 중요한 경고
클래스를 쓴다고 자동으로 객체지향이 되는 것은 아니다.

다음 징후가 보이면 클래스 문법만 쓴 ADT에 가깝다.
- `type`, `kind`, `category` 같은 타입 코드가 있다.
- 메서드 안에 `if`, `switch`, `case`로 타입 분기가 많다.
- 클라이언트가 객체 타입을 검사해서 행동을 결정한다.

### 결론
중요한 것은 `class` 키워드가 아니라, 다형성으로 변경을 흡수하는가다.

---

## 8. 협력이 더 중요하다

타입별로 클래스를 나누고 오퍼레이션을 분배하는 것만으로 완전한 객체지향이 되는 것은 아니다.

- 객체지향에서는 역할, 책임, 협력이 중요하다.
- 객체를 고립시킨 채 로직만 타입별로 나누는 것은 ADT와 클래스의 배치 차이일 뿐이다.
- 결국 객체 설계는 "누가 어떤 책임을 맡고, 누구와 협력하는가"의 문제다.

즉, 이 장의 내용은 3장의 책임 주도 설계와도 연결된다.

## 9. 예제 코드 진화 흐름 한 번에 보기

### 1단계. 기능 분해
- `a_functional_decomposition`
- `b_add_function`
- `c_data_change`
- 특징: 전역 데이터, 메인 함수 중심, 요구사항 변화에 취약

### 2단계. 모듈
- `d_module`
- 특징: 정보 은닉, 네임스페이스 관리, 데이터 변경 영향 축소
- 한계: 타입 차이는 여전히 내부 조건문이 처리

### 3단계. 추상 데이터 타입
- `e_abstract_data+type`
- 특징: 데이터와 연산을 하나의 타입으로 캡슐화
- 한계: 내부 타입 코드와 조건문이 남음

### 4단계. 클래스 기반 객체지향
- `f_class`
- 특징: 타입별 클래스로 분리, 다형성으로 조건문 제거, OCP에 유리

이 장은 갑자기 객체지향을 찬양하는 장이 아니라, 기능 중심 설계가 어디서 무너지고 왜 정보 은닉, 데이터 추상화, 다형성이 필요한지 단계적으로 설득하는 장이다.

## 10. 핵심 정리

| # | 핵심 메시지 |
| --- | --- |
| 1 | 복잡한 시스템은 먼저 분해 기준을 정해야 한다. |
| 2 | 기능 분해는 단순한 문제에는 유효하지만 변화에 약하다. |
| 3 | 모듈은 정보 은닉으로 전역 데이터 문제를 줄이지만 타입 변화까지 해결하진 못한다. |
| 4 | ADT는 데이터와 연산을 묶지만 타입별 행위 차이는 내부 조건문으로 처리하기 쉽다. |
| 5 | 객체지향의 핵심은 클래스 자체가 아니라 다형성으로 변경의 축을 분리하는 데 있다. |
| 6 | 좋은 설계는 현재 모양이 아니라 앞으로 반복될 변경을 기준으로 선택해야 한다. |

## 12. 실무에 연결해서 보기

### 이런 코드가 보이면 7장을 떠올리면 된다
- 서비스 클래스 하나에 모든 분기가 몰려 있다.
- enum 값에 따라 거대한 `switch`가 반복된다.
- UI 입력 처리와 핵심 도메인 계산이 한 메서드에 섞여 있다.
- 데이터 구조 하나 바꾸면 여러 서비스와 유틸을 함께 수정해야 한다.

### 적용 힌트
1. 먼저 감춰야 할 비밀이 무엇인지 찾는다.
2. 변경이 타입 축인지, 연산 축인지 구분한다.
3. 타입 축 변화가 크면 다형성으로 옮긴다.
4. 연산 축 변화가 크면 ADT나 모듈 구조가 더 단순할 수 있다.

## 13. 마무리
7장은 "객체를 잘게 쪼개라"는 장이 아니다.  
"분해 기준을 잘 선택하라"는 장이다.

기능 중심 분해는 출발점이 될 수 있지만, 정보 은닉만으로 충분하지 않을 수 있다. 결국 변경을 가장 잘 흡수하는 단위를 찾아야 하고, 그 결과가 어떤 곳에서는 모듈, 어떤 곳에서는 추상 데이터 타입, 어떤 곳에서는 다형성 기반 객체지향이 된다.

> 설계는 문법이 아니라 변경을 다루는 방식이다.
